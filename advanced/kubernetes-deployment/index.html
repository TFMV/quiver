    Deploying Quiver on Kubernetes¶

    This guide explains how to deploy Quiver on Kubernetes, providing both basic deployment instructions and advanced configuration options.

    Overview¶

    Deploying Quiver on Kubernetes offers several advantages:

    • Scalability: Easily scale your vector database based on demand
    • High Availability: Run multiple replicas for fault tolerance
    • Resource Management: Define precise CPU and memory limits
    • Automated Operations: Leverage Kubernetes for rolling updates, health checks, and auto-scaling
    • Monitoring Integration: Built-in Prometheus metrics support

    Prerequisites¶

    Before you begin, ensure you have:

    • A Kubernetes cluster (v1.19+)
    • kubectl configured to communicate with your cluster
    • Docker for building container images
    • Helm v3 (optional, for Helm chart deployment)

    Quick Start¶

    1. Build the Docker Image¶

    First, build and push the Quiver Docker image:

    # From the root of the Quiver repository
docker build -t your-registry/quiver:latest .
docker push your-registry/quiver:latest

    2. Deploy with kubectl¶

    Create a namespace for Quiver:

    kubectl create namespace quiver

    Apply the Kubernetes manifests:

    kubectl apply -f k8s/pvc.yaml -n quiver
kubectl apply -f k8s/configmap.yaml -n quiver
kubectl apply -f k8s/deployment.yaml -n quiver
kubectl apply -f k8s/service.yaml -n quiver

    Verify the deployment:

    kubectl get pods -n quiver
kubectl get svc -n quiver

    3. Deploy with Helm (Alternative)¶

    If you prefer using Helm:

    helm install quiver ./helm/quiver \
  --namespace quiver \
  --create-namespace \
  --set image.repository=your-registry/quiver \
  --set image.tag=latest

    Configuration¶

    Environment Variables¶

    Quiver can be configured using the following environment variables:

    Variable Description Default
    QUIVER_PORT HTTP server port 8080
    QUIVER_DIMENSION Vector dimension 1536
    QUIVER_MAX_ELEMENTS Maximum number of vectors 1000000
    QUIVER_HNSW_M HNSW hyperparameter M 16
    QUIVER_HNSW_EF_CONSTRUCT HNSW hyperparameter efConstruction 200
    QUIVER_HNSW_EF_SEARCH HNSW hyperparameter efSearch 100
    QUIVER_BATCH_SIZE Number of vectors to batch before insertion 100
    QUIVER_PERSIST_INTERVAL How often to persist index to disk 5m
    QUIVER_BACKUP_INTERVAL How often to create backups 1h
    QUIVER_MAX_BACKUPS Maximum number of backups to keep 5
    QUIVER_BACKUP_COMPRESSION Whether to compress backups true
    QUIVER_LOG_LEVEL Logging level (debug, info, warn, error) info

    ConfigMap¶

    For more complex configurations, use a ConfigMap:

    apiVersion: v1
kind: ConfigMap
metadata:
  name: quiver-config
data:
  quiver.yaml: |
    dimension: 1536
    distance: "cosine"
    max_elements: 1000000
    hnsw:
      m: 16
      ef_construction: 200
      ef_search: 100
    batch_size: 100
    persistence:
      interval: "5m"
      path: "/data/quiver"
    backup:
      interval: "1h"
      path: "/data/backups"
      compression: true
      max_backups: 5
    security:
      encryption_enabled: false
    logging:
      level: "info"
      format: "json"

    Persistence¶

    Quiver requires persistent storage to maintain vector data across pod restarts. The default configuration uses a PersistentVolumeClaim:

    apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: quiver-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi
  storageClassName: standard

    Storage Class

    Choose an appropriate storageClassName for your cluster. For production, consider using SSD-backed storage for better performance.

    Scaling¶

    Horizontal Pod Autoscaler¶

    Quiver can be scaled horizontally for read-heavy workloads. Use a Horizontal Pod Autoscaler (HPA) to automatically scale based on CPU and memory usage:

    apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: quiver-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: quiver
  minReplicas: 1
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80

    Read-Only Replicas

    When scaling Quiver horizontally, consider using read-only replicas for search operations, with a single primary instance handling writes.

    Resource Limits¶

    Set appropriate resource requests and limits for your workload:

    resources:
  requests:
    cpu: 500m
    memory: 512Mi
  limits:
    cpu: 2
    memory: 4Gi

    Adjust these values based on your vector dimension and dataset size.

    Monitoring¶

    Prometheus Metrics¶

    Quiver exposes Prometheus metrics at the /metrics endpoint. The following metrics are available:

    • quiver_search_requests_total: Total number of search requests
    • quiver_search_latency_seconds: Search request latency in seconds
    • quiver_vector_count: Number of vectors in the index
    • quiver_memory_usage_bytes: Memory usage in bytes

    ServiceMonitor¶

    If you're using the Prometheus Operator, you can configure a ServiceMonitor:

    apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: quiver
  labels:
    app: quiver
spec:
  selector:
    matchLabels:
      app: quiver
  endpoints:
  - port: http
    path: /metrics
    interval: 15s
    scrapeTimeout: 10s

    Health Checks¶

    Quiver provides health check endpoints for Kubernetes probes:

    livenessProbe:
  httpGet:
    path: /health/live
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10
readinessProbe:
  httpGet:
    path: /health/ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5

    Security¶

    Network Policies¶

    Restrict network access to your Quiver deployment with a NetworkPolicy:

    apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: quiver-network-policy
spec:
  podSelector:
    matchLabels:
      app: quiver
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: client
    ports:
    - protocol: TCP
      port: 8080

    Secrets Management¶

    For sensitive configuration (like encryption keys), use Kubernetes Secrets:

    apiVersion: v1
kind: Secret
metadata:
  name: quiver-secrets
type: Opaque
data:
  encryption-key: <base64-encoded-key>

    Then mount the secret in your deployment:

    env:
- name: QUIVER_ENCRYPTION_KEY
  valueFrom:
    secretKeyRef:
      name: quiver-secrets
      key: encryption-key

    Troubleshooting¶

    Common Issues¶

    Pod Fails to Start¶

    Check the pod logs:

    kubectl logs -f deployment/quiver -n quiver

    Verify resource limits are appropriate for your dataset size.

    Persistence Issues¶

    Check the PVC status:

    kubectl get pvc -n quiver
kubectl describe pvc quiver-pvc -n quiver

    Ensure the storage class exists and has provisioner support.

    Performance Problems¶

    1. Check resource utilization:
    kubectl top pods -n quiver

    1. Review Prometheus metrics for bottlenecks.

    2. Adjust HNSW parameters for your specific workload.

    Helm Chart Reference¶

    If you're using the Helm chart, here are some common configuration options:

    # Basic installation
helm install quiver ./helm/quiver --namespace quiver --create-namespace

# Custom resource configuration
helm install quiver ./helm/quiver \
  --namespace quiver \
  --set resources.requests.memory=1Gi \
  --set resources.limits.memory=8Gi

# Enable autoscaling
helm install quiver ./helm/quiver \
  --namespace quiver \
  --set autoscaling.enabled=true \
  --set autoscaling.minReplicas=2 \
  --set autoscaling.maxReplicas=10

# Custom storage configuration
helm install quiver ./helm/quiver \
  --namespace quiver \
  --set persistence.storageClass=fast-ssd \
  --set persistence.size=50Gi

# Enable Prometheus ServiceMonitor
helm install quiver ./helm/quiver \
  --namespace quiver \
  --set metrics.serviceMonitor.enabled=true

    Conclusion¶

    Deploying Quiver on Kubernetes provides a robust, scalable, and manageable vector database solution. By leveraging Kubernetes features like auto-scaling, health checks, and persistent storage, you can run Quiver in production with confidence.

    For more advanced configurations and optimizations, refer to the Performance Tuning guide.
